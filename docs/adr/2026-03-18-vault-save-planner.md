# 2026-03-18 Vault Save Planner

## 背景

当前插件在执行“保存到本地”这类请求时，会把 Vault 根目录作为 `workingDirectory` 交给 Codex SDK。由于没有目录规划层，agent 很容易直接把文件写到 Vault 根目录。

这和用户对 Obsidian 知识库的实际使用方式不一致。用户希望插件像 Claudian 一样：

1. 优先参考 Vault 根目录的 README、索引、指南等规范文件
2. 如果规范不明确，再结合目录结构和已有文件命名分布推断
3. 在多个候选目录中，优先选择与待保存内容类型最匹配的目录
4. 只有兜底时才落到当前笔记同级目录或 Vault 根目录

## 备选方案

### 方案 A：纯 prompt 提示

做法：

- 只在 prompt 里写一条静态提示，例如“默认保存到当前笔记同级目录”

优点：

- 实现最简单

缺点：

- 无法利用 Vault 的组织规范
- 不可解释，也不稳定
- 仍然会频繁落到根目录

### 方案 B：规则驱动 planner + prompt 注入

做法：

- 插件先读取少量规范文件、目录结构样本和最近待保存内容
- 通过纯逻辑 planner 计算推荐目录、原因和回退链
- 再把结果明确注入 prompt，指导 Codex 保存

优点：

- 可解释、可测试
- 不需要改写 Codex SDK 的底层工具调用
- 能明显改善默认保存目录行为

缺点：

- 需要维护一套规则和轻量扫描逻辑
- 仍属于“引导式”而不是“硬拦截式”

### 方案 C：底层写文件拦截/重写

做法：

- 在 agent 真正写文件时拦截目标路径，强制改写到插件计算出的目录

优点：

- 约束最强

缺点：

- 风险高
- 会侵入 SDK 或工具链
- MVP 阶段实现成本和不确定性都过大

## 决策

采用方案 B：

- 新增纯逻辑 `vault-save-planner`
- `chat-view.ts` 负责采集：
  - Vault 根目录规范文件
  - 目录结构和文件名样本
  - 当前笔记路径
  - 最近一条 assistant 内容
- `context-builder.ts` 负责把 planner 结果写成明确的保存提示

固定决策顺序：

1. 规范文件
2. 目录结构和命名分布
3. 内容类型匹配
4. 当前笔记同级目录
5. Vault 根目录

## 影响

- “保存到本地”请求的默认行为不再是静态规则，而是依赖 Vault 结构分析
- prompt 中会出现推荐目录、原因、置信度和回退链
- 不改变当前 `workingDirectory = Vault root` 的线程工作目录设计

## 后续

- 如果后续需要更强约束，再评估是否为写文件操作增加插件级拦截或确认层
- 如果发现规则覆盖不足，再扩展内容类型和规范文件解析能力
