# 2026-03-20 Composer Attachments

## 状态

已采纳

## 背景

聊天输入区原本只支持纯文本，附加上下文完全依赖“当前笔记 + 当前选区”的自动快照。这种方式虽然简单，但无法覆盖两个高频需求：

- 用户希望显式指定额外的 Vault 文件，而不是通过切换活动笔记来表达意图
- 用户希望把截图或图片直接粘贴到对话里参与本轮请求

如果分别为 `@引用` 和图片粘贴设计两套独立状态和发送逻辑，`chat-view.ts` 会快速演化成难以维护的 UI 状态机。

## 决策

采用统一的“输入区附件”模型，把显式文件引用和图片粘贴都视为 `ComposerAttachment`：

- `vault-file`
  - 只记录 Vault 相对路径
  - 在发送瞬间读取文件内容快照
- `pasted-image`
  - 写入 Vault 配置目录下的插件缓存目录 `plugins/codexian/.cache/pasted-images/`
  - 在 prompt 中传递相对路径和元信息，不直接内联图片内容

输入区继续使用普通 `textarea`，不升级为富文本编辑器：

- `@` 只作为文件选择触发器
- 选中文件后，从正文移除 `@query`，改为在输入区下方显示 attachment chip
- 图片通过 `paste` 事件捕获，落盘后也显示为 attachment chip

发送链路保持不变：

- `chat-view.ts` 负责维护附件状态
- `context-builder.ts` 负责把附件转换为稳定的结构化文本上下文
- `codex-service.ts` 仍然只接收单字符串 prompt

## 影响

正向影响：

- `@引用` 和图片粘贴共用同一套状态、去重、移除、预算和清理逻辑
- 不需要在当前阶段引入新的多模态 SDK 抽象
- UI 和上下文拼装职责仍然清晰

约束与取舍：

- 首期只支持 Vault 内 Markdown 文件，不支持外部目录、MCP 或 provider 化引用系统
- 图片只支持粘贴，不支持拖拽和外链
- 图片能力依赖 Vault 内缓存文件路径，而不是原生图片消息协议
