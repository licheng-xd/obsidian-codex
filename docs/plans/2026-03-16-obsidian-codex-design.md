# Obsidian Codex 插件设计文档

日期：2026-03-16

## 1. 背景

项目目标是在 Obsidian 中实现一个 Codex 插件，参考 `Claudian` 的产品方向，但首个版本不追求完整代理能力，而是优先构建稳定、清晰、可验证的聊天交互基础。

本项目中的 `Codex` 明确定义为 OpenAI Codex。首版后端采用 OpenAI 官方 `@openai/codex-sdk` 控制本地 Codex agent，而不是 Claude Code，也不是泛指任意代码助手。

用户已确认首个版本采用如下范围：

- 聊天侧边栏
- 当前笔记上下文注入
- 当前选中文本上下文注入

在首版落地后，当前实现已增加两项输入区附件能力：

- Vault 内 Markdown 文件的显式 `@引用`
- 粘贴图片附件

## 2. 设计目标

- 在 Obsidian 中提供一个稳定的聊天侧边栏作为统一入口。
- 在用户发送消息时，自动注入当前活动笔记和编辑器选中文本的上下文快照。
- 保持 UI、上下文构建与 Codex 调用边界清晰，但首版不为未来多后端预留额外抽象层。
- 保持 MVP 范围克制，不做内联编辑、自动写回或复杂权限编排。
- 输入区允许显式附件，但仍不扩展到外部目录、MCP 或 provider 化上下文系统。

## 3. 非目标

- 首版不实现完整的 Claudian 功能对齐。
- 不实现 provider 化多源 `@mention`、外部目录引用、MCP、技能系统、自定义代理。
- 首版不支持移动端。
- 首版不默认开放自动写文件、自动执行危险命令等高风险能力。
- 首版不为了“未来可能的第二种后端”提前设计 Adapter 抽象层。

## 4. 方案对比

### 方案 A：聊天侧边栏 + 当前笔记/选区上下文注入

这是本次选定方案。

优点：

- 实现成本低，能快速建立可用 MVP。
- 交互路径清晰，便于验证 Codex 在 Obsidian 中的核心价值。
- 上下文来源明确，安全边界相对容易控制。
- 后续可自然演进到文件引用、内联编辑、更多上下文源。

缺点：

- 首版能力有限，不具备完整代理体验。
- 用户需要切换活动笔记或显式选区来表达上下文意图。

### 方案 B：聊天侧边栏 + 当前笔记/选区 + 多文件引用

优点：

- 更接近知识管理场景中的真实使用需求。
- 能更早验证跨文件分析能力。

缺点：

- 需要提前设计文件检索、引用 UI、上下文拼装和截断策略。
- 会显著增加范围，不利于当前空仓库状态下的快速落地。

### 方案 C：接近 Claudian 的完整代理式插件

优点：

- 产品完成度高，长期上限更强。

缺点：

- 范围过大，涉及命令执行、权限模型、文件写回、复杂上下文、更多 UI 状态。
- 不适合当前项目启动阶段，会显著拉高不确定性和返工概率。

结论：首版采用方案 A 起步；后续在不改动核心会话链路的前提下，已把方案 B 中最小且高价值的“显式文件引用”收敛为输入区附件能力。

## 5. 已解决的核心决策

- 后端：OpenAI Codex SDK
- 会话模型：单侧边栏单线程，先以内存线程为主，清空会话即创建新线程
- 流式响应：使用 `thread.runStreamed()`
- 取消机制：每轮请求单独创建 `AbortController`
- 历史管理：依赖 Codex thread 自身的会话语义，不在 MVP 中手工重建历史上下文

## 6. MVP 架构

首版按最小文件数落地，不做多余分层：

```text
src/
  main.ts
  codex-service.ts
  context-builder.ts
  chat-view.ts
  settings.ts
  settings-tab.ts
```

### 模块职责

- `main.ts`
  - 注册侧边栏视图、命令、设置页。
  - 持有插件级设置与 `CodexService`。

- `codex-service.ts`
  - 封装对 `@openai/codex-sdk` 的直接调用。
  - 管理单个 `Codex` 实例、当前 `Thread`、当前轮次的 `AbortController`。
  - 把 SDK 事件转换成侧边栏可消费的简单消息事件。

- `context-builder.ts`
  - 从当前活动笔记和选区生成发送时快照。
  - 同时执行截断和 token budget 控制。

- `chat-view.ts`
  - 同时承载视图 UI 和交互控制。
  - 处理发送、取消、流式渲染、上下文提示和错误展示。

- `settings.ts`
  - 默认设置、设置类型、序列化辅助。

- `settings-tab.ts`
  - 渲染插件设置页。

## 7. 上下文模型

发送消息时构造一次上下文快照：

- `activeNote`
  - 文件路径
  - 标题
  - 经过预算控制后的正文片段
  - 可选的 frontmatter 摘要

- `selection`
  - 是否存在选中文本
  - 选中文本内容
  - 选区在文档中的起止位置信息

- `uiState`
  - 当前时间
  - 当前 Vault 名称
  - 需要时可附加只读元信息

- `attachments`
  - `vault-file`
    - Vault 相对路径
    - 发送瞬间读取的文件内容快照
  - `pasted-image`
    - Vault 相对路径
    - `mimeType / sizeBytes / dimensions`

设计约束：

- 上下文只在发送瞬间快照一次，避免响应过程中受到编辑器继续变动的影响。
- 当无活动 Markdown 编辑器时，允许仅发送纯聊天消息。
- 当无选区时，`selection` 字段为空，不阻塞请求。
- 不复用 Claudian 的 XML 标签格式；首版使用简单、稳定的文本分段拼装。
- 显式 `@引用` 的优先级高于自动附加的当前笔记正文。
- 图片附件不以内联 base64 形式注入，而是通过 Vault 内缓存路径和元信息传给 Codex。

## 8. Token Budget 与截断策略

首版必须在发送前控制上下文体积。

### 基本策略

- 不手工回放全部历史消息。
- 会话连续性由 Codex thread 维持，因此本插件只控制“本轮额外注入的编辑器上下文”。
- 本轮上下文注入总预算默认控制在约 `6k` tokens 以内。

### 优先级

1. 用户当前输入
2. 当前选中文本
3. 显式 `@引用` 文件
4. 当前笔记摘要
5. 图片附件元信息
6. 可选元信息

### 截断规则

- 如果选区存在：
  - 先保留选区全文，预算上限约 `1.5k` tokens。
  - 当前笔记正文不再重复包含与选区重叠的部分。
- 显式 `@引用` 文件：
  - 单轮最多 5 个文件
  - 单文件按保守字符预算截断
  - 若已显式引用当前活动笔记，则不再自动重复注入 active note 正文
- 图片附件：
  - 不注入图片二进制
  - 只注入缓存路径与元信息
- 当前笔记在剩余预算内按以下顺序抽取：
  - 文档开头
  - 选区附近窗口
  - 文档结尾
- 当前笔记正文预算上限约 `4k` tokens。
- 额外保留约 `500` tokens 给路径、标题、标签式说明和提示语包装。

### 实现约束

- Phase 3 开始时即接入实际 token 计数方案，不接受“全文直接塞入请求”。
- 如果首轮未接入精确 tokenizer，则必须使用保守字符预算作为临时保护，且不能把它当作长期方案。

## 9. 交互流程

### 9.1 打开侧边栏

1. 用户点击命令或侧边栏图标打开 Chat View。
2. 如果当前没有线程，则 `CodexService` 延迟创建新 thread。
3. 侧边栏显示当前活动笔记名与选区状态，作为将被注入的上下文提示。

### 9.2 发送消息

1. 用户在输入框输入消息并发送。
2. `chat-view.ts` 调用 `context-builder.ts` 获取当前上下文快照。
3. `codex-service.ts` 将用户输入与上下文快照拼成一轮请求。
4. 服务调用 `thread.runStreamed()`。
5. UI 增量渲染助手回复。
6. 当前轮结束后保留 thread 以便下一条消息继续同一会话。

### 9.3 取消请求

- 用户点击取消按钮。
- `chat-view.ts` 调用当前轮的 `AbortController.abort()`。
- 当前助手消息标记为“已中断”。

### 9.4 切换笔记或修改选区

- 不自动重放上一条请求。
- 仅更新 UI 中“当前上下文”提示。
- 下一次发送消息时重新生成快照。

## 10. UI 设计

侧边栏 MVP 包含以下区域：

- 标题栏
  - 插件名称
  - 当前上下文状态提示
  - 可选的清空会话按钮

- 消息区
  - 用户消息
  - 助手消息
  - 流式响应中的中间态
  - 错误消息块

- 输入区
  - 多行输入框
  - `@引用` 候选下拉
  - 文件 / 图片附件 chips
  - 发送按钮
  - 当前上下文摘要，例如“当前笔记：X / 已选中 132 字”

不在首版加入：

- 多 tab 会话
- slash commands
- diff 预览
- 文件引用面板
- 多会话列表

## 11. 错误处理与安全策略

### 错误处理

- 无活动笔记：允许继续聊天，但 UI 明确提示本次未注入笔记上下文。
- 无选区：正常发送，仅省略选区上下文。
- 本地 Codex CLI 或 SDK 不可用：展示明确错误，并保留用户输入以便重试。
- 流式请求中断：将本轮响应标记为中断状态，允许用户重新发送。
- thread 创建失败：退回到未连接状态，要求用户修复本地环境后重试。

### 安全策略

- 首版只做上下文读取，不做自动写回。
- 上下文来源限制为当前活动笔记、当前编辑器选区、显式 `@引用` 文件和粘贴图片附件。
- 不默认开放任意外部路径作为上下文源。
- 使用 SDK 的 `workingDirectory` 指向 Vault 根目录。
- 首版显式启用保守的 sandbox / approval 配置，不以 `danger-full-access` 为默认值。
- 对非 Git Vault，需要显式设置 `skipGitRepoCheck`，并在文档中标明这是 Obsidian 场景下的特例。

## 12. 测试策略

### 单元测试

- `context-builder.ts`
  - 有活动笔记 + 有选区
  - 有活动笔记 + 无选区
  - 无活动编辑器
  - 超长笔记被正确截断
  - 选区优先级高于正文摘录

- `codex-service.ts`
  - SDK 事件被映射为 UI 可消费事件
  - 中断/失败状态处理正确

### 集成验证

- 插件加载后可注册侧边栏视图和命令。
- 打开笔记并选中文本后，发送消息时能看到上下文摘要变化。
- 后端返回流式内容时，消息区能逐步更新。

### 手工验证

- 在桌面 Obsidian 中安装开发版插件。
- 验证以下场景：
  - 有活动笔记且有选区
  - 有活动笔记但无选区
  - 无活动 Markdown 编辑器
  - 后端报错或超时

## 13. 已确认决策

- 首个版本采用聊天侧边栏作为唯一主入口。
- 上下文注入仅包含当前笔记和当前选中文本。
- 发送时使用上下文快照，而不是持续绑定编辑器对象。
- 首版后端采用 OpenAI Codex SDK。
- 首版采用单侧边栏单线程模型。
- 首版不做自动写回和复杂代理能力。

## 14. 仍待验证的问题

- Obsidian 桌面运行时与当前 SDK 版本的兼容性。
- 是否需要在首版持久化 thread id 以支持重开 Obsidian 后恢复对话。
- 默认 sandbox / approval 组合在 Vault 使用场景下的可用性。
- 子进程架构风险：`@openai/codex-sdk` 底层需要在 Obsidian Electron 环境中拉起本地 `codex` CLI。需要验证 `spawn`、PATH/环境变量继承、stdio 流式读取和中断信号在插件运行时是否稳定。

实现阶段的第一个技术验证点：

- 在 Obsidian Electron 环境中验证 `spawn codex CLI` 的可行性。
- 这项验证未通过前，不继续投入完整聊天 UI 开发。
