# Codexian 三阶段演进路线图

## 背景

对比 `YishenTu/claudian` 之后，`Codexian` 当前最大的短板不是“没有更多 provider”，而是它还主要停留在“单侧栏聊天插件”，没有形成稳定的 agent 工作台体验。

当前仓库已经具备：

- 单侧栏聊天
- 当前笔记与选区上下文注入
- Vault 内 Markdown 文件 `@引用`
- 粘贴图片附件
- 最近会话恢复
- 基于 `@openai/codex-sdk` 的稳定本地执行链路

但离更顺手的日常使用还有三类差距：

1. 会话流转不够顺滑
2. 编辑闭环不够短
3. 上下文入口不够像“工作上下文系统”

## 规划原则

1. 继续坚持单 provider、桌面优先、本地优先，不为了未来抽象提前 provider 化。
2. 优先补最影响体感的能力，而不是优先堆最显眼的功能名词。
3. 每个迭代都要求：
   - `npm test`
   - `npm run typecheck`
   - `npm run build`
4. 每个迭代都要继续控制模块边界，避免把新逻辑继续堆进 `chat-view.ts`。

## 迭代 1：会话工作台化

### 目标

把当前“单聊天页 + 历史弹层”升级成“单侧栏会话工作台”，优先解决长期使用时的新建、恢复、切换体验，以及 `chat-view.ts` 持续膨胀的问题。

### 范围

- 在当前单侧栏模型内，补齐更明确的会话操作：
  - 新会话
  - 恢复最近会话
  - 当前会话状态可见
- 把当前历史弹层扩成更明确的 session workbench 入口，而不只是被动列表
- 将 `chat-view.ts` 的职责拆分为：
  - 会话工作台状态
  - 发送/取消与 thread 生命周期
  - 消息渲染
  - composer 相关 UI 协调
- 增加对应命令入口，降低用户必须点 UI 的依赖

### 明确不做

- 不做多 tab
- 不做 provider registry
- 不做外部目录上下文
- 不做 MCP
- 不做 inline edit
- 不承诺“真 fork 会话”

### 为什么先做

- 这是最直接影响“用久了是否顺手”的一层
- 这是后续 inline edit 和更复杂上下文系统的地基
- 这是控制 `chat-view.ts` 复杂度继续失控的必要步骤

### 验收标准

1. 用户能清晰区分“继续当前会话”和“开启新会话”
2. 恢复会话是显式工作流，不只是历史列表点击
3. `chat-view.ts` 不再同时承担全部状态管理、thread 生命周期和消息渲染
4. 文档与命令入口能准确说明当前会话模型

## 迭代 2：补上 Inline Edit

### 目标

把插件从“聊天助手”推进到“笔记改写助手”，缩短 Obsidian 内的编辑闭环。

### 范围

- 增加 inline edit 命令
- 支持两种入口：
  - 选中文本改写
  - 光标处生成/插入
- 使用最小可用的 diff / 确认机制
- 写回仍然受显式用户确认控制

### 明确不做

- 不做多段批量编辑
- 不做自动写回全文
- 不做复杂 patch 编辑器
- 不做跨文件联动编辑

### 为什么第二个做

- 这是 Claudian 体感优势最明显的单点能力
- 它直接贴合 Obsidian 的核心工作流
- 它不要求先把 provider、多目录、MCP 全部做出来

### 验收标准

1. 用户可从编辑器直接发起一次局部改写
2. 改写结果在应用前可检查
3. 不破坏现有聊天主链路
4. 纯逻辑和 diff 组装部分有新增测试

## 迭代 3：把上下文系统升级为工作上下文系统

### 目标

把当前“单轮附件”升级为“会话级可复用工作上下文”，增强 agent 感，但继续控制风险。

### 范围

- 在现有 `@文件 + 粘贴图片` 基础上，引入“持久上下文集合”
- 第一阶段只支持 Vault 内目录、固定文件集合或固定文档列表
- 增加轻量命令入口，例如：
  - `/new`
  - `/resume`
  - `/clear-context`
- 让上下文不需要每轮手动重新指定

### 明确不做

- 不直接开放外部目录
- 不做 MCP server 管理
- 不做 skills 系统
- 不做 provider 化 mention 系统

### 为什么放第三个做

- 这是工作流能力增强最明显的一层，但复杂度也最高
- 没有迭代 1 的状态拆层，这一层会继续把 `chat-view.ts` 压垮
- 没有迭代 2 的编辑闭环，工作上下文再强也会停留在“看和聊”

### 验收标准

1. 用户能在会话级复用一组上下文
2. 上下文来源和权限边界仍然清晰
3. 命令入口明显增强 agent 工作感
4. README 与设计文档明确记录新的上下文边界

## 顺序依赖

### 先做迭代 1

因为它解决的是结构和工作流入口问题。

### 再做迭代 2

因为它解决的是编辑闭环问题，而且依赖较少。

### 最后做迭代 3

因为它对状态管理、输入区和会话语义的要求最高，应该建立在前两轮的基础上。

## 当前建议

当前最值得立即启动的是“迭代 1：会话工作台化”。原因不是它最炫，而是它同时解决两类问题：

1. 用户操作路径不顺
2. 当前代码结构继续扩展的成本过高

后续所有增强项，都应该建立在这轮拆层之后再继续推进。
